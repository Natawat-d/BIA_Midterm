Ch.1 · Overview of BI, Analytics & Data Science

Key idea · BI (Business Intelligence) is the modern evolution of DSS; its job is to turn data into managerial action, and every method is descriptive, predictive or prescriptive.

Need for DSS (Decision Support System) — faster computing + big-data processing (hardware/software/network growth); collaboration (share dashboards & reports); better data management (databases + data warehouses organise enterprise data); big data (large structured + unstructured datasets); analytical tools (dashboards, OLAP, data mining); human cognitive limits (let systems analyse large data); knowledge management (store organisational experience); anywhere support (cloud + mobile). Purpose: transform data → information → insight → decision.

Relationship (evolution) — managers couldn't analyse exploding data by hand, so tools evolved: DSS (1970s) = first data + model decision aids → BI (1990s, term popularised by Gartner) = DSS scaled up with data warehouses, OLAP & dashboards → BA (Business Analytics) = the statistical / mining / optimization techniques applied inside BI. Data science extends this with AI/ML on big + unstructured data.

DSS — interactive system that pairs data + models to tackle semi/unstructured problems (no fixed rule or formula, so human judgment is needed) — Gorry & Scott-Morton 1971. Aim: support, not replace the manager → raise decision quality. 4 components: data mgmt (the database) · model mgmt (the analytical models) · knowledge subsystem (org rules/expertise) · UI (the dialog) — plus the user.

Simon's phases (1977) — Intelligence (spot the problem/opportunity, gather data) → Design (build models, generate alternatives) → Choice (evaluate & pick the best) → Implementation (act) → Monitoring (check results, loop back if wrong). It is iterative — you can return to any earlier phase. Car eg: car too old → compare sedan/SUV/EV → pick affordable+efficient → buy → track fuel/maintenance.

Gorry–Scott-Morton grid — classifies decisions by structure (can you write a rule? structured / semi / unstructured) × control level (time horizon: operational day-to-day / managerial / strategic long-term). Structured-operational = order entry, payroll (automate); semi = credit approval, scheduling (rules + judgment); unstructured-strategic = mergers (M&A), R&D (mostly judgment). Operational → more structured; strategic → more unstructured — DSS targets the semi/unstructured cells.

BI — an umbrella / "content-free" term (means different things to different firms): architectures + databases + tools + methods + apps. Goal: give managers easy, timely data access so they analyse & decide. Value chain Data → Information → Knowledge → Decisions → Actions. 4 components: data warehouse (stores) · business analytics (analyses) · BPM (monitors performance) · UI (presents) — future = embedded/intelligent systems. Flow: sources → DW (Data Warehouse) → analytics → performance mgmt → UI.

OLTP vs OLAP — OLTP (Online Transaction Processing) = operational systems that capture day-to-day transactions (ERP / SCM / CRM / POS); highly normalized for fast, safe writes. OLAP (Online Analytical Processing) = analysis on the warehouse; multidimensional cubes for fast reads & decision support. (ERP=Enterprise Resource Planning, SCM=Supply Chain Management, CRM=Customer Relationship Management, POS=Point of Sale.)

Big Data — 3 Vs: Volume (sheer size, TB→PB→ZB), Velocity (speed of generation/flow — batch → streaming), Variety (mix of structured + semi/unstructured: text, image, sensor, clickstream); some add Veracity (trust/quality) + Value (worth once analysed). Analytics ecosystem (≈11 player types) — who makes the analytics economy work: data-generation providers (sensors / IoT = Internet of Things); data-management providers (databases / cloud); data-warehouse providers; middleware providers (integration); analytics-software developers (BI tools); application developers (industry apps); data-service providers; analysts/influencers; academic institutions (education/certification); regulators (data governance); user organisations (the businesses that apply it).

Ch.3 · Descriptive Analytics I — Data, Statistics & Visualization

Key idea · Descriptive analytics = "what happened?"; first know the nature of data (types/scales) and clean it, then summarise (statistics) and communicate (visualization).

Nature of data — raw facts from observation/experiment (numbers, text, images, audio); the lowest abstraction → information (organised data) → knowledge (actionable) → decisions. Quality & integrity are critical ("garbage in → garbage out"). Analytics-ready data should meet quality metrics: source reliability, content accuracy, accessibility, security/privacy, richness, consistency, timeliness (currency), granularity (level of detail), validity/relevance.

Scales of measurement — the scale fixes which stats & charts are valid. Nominal = labels only (gender, country — count, can't order). Ordinal = ordered but unequal gaps (rankings, Likert). Interval = equal gaps, no true zero (°C — can't say "twice as hot"). Ratio = equal gaps + true zero (income, age, weight — ratios meaningful).

Scale eg · jersey number = nominal; rank 1–5 = ordinal; °C = interval (20° ≠ 2×10°); kg = ratio (true 0 → 20 kg = 2×10 kg).

Preprocessing (4 tasks) — typically ~80% of the work. Consolidation (access, select & integrate sources into one dataset) → Cleaning (impute missing via mean/median/mode, remove noise/outliers, fix errors) → Transformation (normalize 0–1, binning, derive new attributes, encode categories) → Reduction (cut columns via PCA = Principal Component Analysis / correlation; cut rows via sampling; balance class skew via over/under-sampling).

Statistics — Descriptive (summarise the data you have) vs Inferential (generalise from a sample to a population). Central tendency (typical value): mean / median / mode. Dispersion (spread): range (max−min), variance, standard deviation (SD), quartiles, box-plot. Shape: histogram, skewness (asymmetry — which tail is longer), kurtosis (peakedness / tail weight).

Regression — models input x → output y to explain each predictor's effect and forecast y; y=β₀+β₁x; fit by OLS (Ordinary Least Squares = minimise squared error). Steps: assess → fit → evaluate → deploy. Quality: R² (% variance explained), p-value (significance), error MSE/MAD/RMSE. 5 assumptions: linearity, independence, normality, constant variance, no multicollinearity. Correlation = strength/direction only (no prediction). Logistic regression = classification: sigmoid squashes output to a 0–1 probability.

Business reporting — a report = a communication artifact that drives decisions; data pulled from internal + external systems (via ETL). Functions: operational control, inform, present analysis, persuade, build organizational memory. Types: metric-management (track KPI = Key Performance Indicator / SLA = Service Level Agreement), dashboard (graphical one-page summary), balanced-scorecard (financial / customer / internal / learning).

Visualization = "insight, not pictures" (Shneiderman) — the goal is understanding, not decoration. Data viz = show raw values; information viz = summarised/contextual for storytelling. Visual analytics = information viz + predictive analytics (combine "what happened" with "what will / why", interactively). Ch.4 · Descriptive Analytics II — Data Warehousing & BPM

Key idea · The data warehouse (DW) is BI's cleansed, integrated, historical backbone; ETL feeds it, OLAP makes it fast, BPM (Business Performance Management) / scorecards turn analysis into managed action.

Data warehouse — a cleansed, integrated store built for analysis, not transactions. 4 traits: subject-oriented (organised by business subject e.g. sales, not by app), integrated (one consistent format from many sources), time-variant (keeps history for trends), non-volatile (load once, rarely update/delete). Summarised, NOT normalized (denormalised for fast reads), metadata-rich. Data mart = a department-sized subset: dependent (from the EDW) / independent (standalone). ODS (Operational Data Store) = staging area; EDW (Enterprise DW) = org-wide; metadata = "data about data" (structure/origin/usage; aids governance).

Framework & architecture — pipeline: sources → ETL → EDW → data marts → analytics (OLAP / dashboards / mining / reporting). 3-tier (Tier 1 client UI → Tier 2 application/OLAP server → Tier 3 database server) separates layers for scalability vs 2-tier (combined; simpler, less scalable). 5 architectures: independent marts · data-mart bus (conformed/shared dimensions) · hub-and-spoke (central EDW + dependent marts) · centralized · federated (logically integrate existing systems). Choose by: info-sharing needs, mgmt requirements, urgency, end-user tasks, resources, strategic vision, IT compatibility, staff skills, complexity, politics.

Integration — ETL (Extract-Transform-Load) is the integration backbone: pull from sources, cleanse/conform, load to DW. EAI (Enterprise Application Integration) pushes operational data source→DW; EII (Enterprise Information Integration) gives a real-time unified view across sources. Build approach: Inmon top-down (enterprise DW first, then marts — slow, consistent) vs Kimball bottom-up (dimensional marts first, integrate via conformed dimensions — fast, incremental).

Dimensional modeling — optimised for analytical queries: fact tables (numeric measures + foreign keys) surrounded by dimension tables (descriptive context). Star = one central fact + flat dimensions (simplest, fastest). Snowflake = dimensions normalized into sub-tables (less redundancy, more joins/complexity). Multidimensional analysis = dimensions (product / region / time / salesperson) × measures (revenue / profit) = the OLAP cube.

OLAP operations (navigate the cube) — Slice (fix 1 dimension) · Dice (subset on several) · Drill down/up (detail ↔ summary) · Roll-up (aggregate higher) · Pivot (rotate axes). DW vs Data Lake — DW = structured, schema-on-write, for business analysts; Data Lake = raw/native data, schema-on-read, cheaper storage, for data scientists / big-data.

BPM (Business Performance Management) closed loop: Strategize (vision/goals) → Plan (budgets/targets) → Monitor/Analyze (measure with KPIs) → Act/Adjust; raises real-time alerts on risks/opportunities. KPI = a metric tied to a goal: outcome / lagging (result, e.g. revenue) vs driver / leading (predictor, e.g. sales leads); good KPIs have targets + benchmarks + timeframe + strategic alignment.

Balanced Scorecard (BSC) (Kaplan&Norton 1992) — turns strategy into 4 linked perspectives joined by a cause-effect strategy map: Financial · Customer · Internal Process · Learning&Growth. Six Sigma — reduce process defects toward 3.4 DPMO (Defects Per Million Opportunities) via DMAIC (Define-Measure-Analyze-Improve-Control). BSC = strategic management framework; Six Sigma = process-improvement method. Good measurement: focus on key success factors; mix past/present/future; balance stakeholders; align top→ops; set realistic targets.

Ch.5 · Predictive Analytics I — Data Mining

Key idea · Data mining (DM) is a disciplined process (CRISP-DM), not a crystal ball — find valid, novel, useful, understandable patterns in structured data: predict (classify/regress) & group (cluster/associate).

Data mining (Fayyad 1996) — the nontrivial discovery of valid, novel, useful, understandable patterns in large data; a.k.a. knowledge discovery. Blends statistics + ML (Machine Learning) + AI + databases + visualization, usually on a DW, often run by end-users. DM ≠ statistics: stats tests a hypothesis on a sample; DM discovers unknown patterns in huge data. Why now: fierce competition, recognised data value, big data, data warehouses, cheap storage/compute, data-driven culture.

How DM works — finds mathematical relationships = patterns: association · prediction · cluster · sequential/time-series. 8 tasks: prediction, classification (assign to known classes), regression (predict a number), clustering (group without labels), association (co-occurrence), link analysis, sequence analysis, outlier/anomaly. Applied in: CRM (churn, cross-sell), banking (fraud, credit risk), retail (inventory, placement), manufacturing (predictive maintenance), insurance, stock forecasting, healthcare diagnosis, crime/government.

Two key contrasts — Classification vs Regression: categorical vs numeric output (both supervised). Classification vs Clustering: supervised (you have a labelled target) vs unsupervised (no target — let groups emerge). Market-basket analysis = which items are bought together (diaper+beer) → store layout / cross-sell / promotion. (Algos above: DT=Decision Tree, ANN=Artificial Neural Network, SVM=Support Vector Machine, kNN=k-Nearest Neighbor, NB=Naïve Bayes, SOM=Self-Organizing Map.)

Processes (make DM systematic, not trial-and-error) — CRISP-DM (Cross-Industry Standard Process for DM, 6 iterative phases): Business Understanding → Data Understanding → Data Prep → Modeling → Evaluation → Deployment; the prep-heavy front end ≈ 85% of effort. SEMMA (SAS: Sample-Explore-Modify-Model-Assess). KDD (Knowledge Discovery in Databases): select → clean → transform → mine → interpret.

Classification = supervised, predicts a categorical class (spam/credit/diagnosis). Confusion matrix (TP/FP/TN/FN) → Accuracy=(TP+TN)/all (misleading if imbalanced); Precision=TP/(TP+FP) (of predicted-positives, how many right); Recall/TPR=TP/(TP+FN) (of actual-positives, how many caught); TNR=TN/(TN+FP). Validation (estimate real-world accuracy): holdout 70/30; k-fold CV (Cross-Validation); leave-one-out; bootstrap. ROC/AUC: plots TPR vs FPR; AUC 0.5=chance, 1=perfect — good for skewed data.

Eg · TP80, FP20, FN10, TN90 → Accuracy 85%, Precision 80%, Recall ≈89%.

Decision tree — divide-and-conquer: recursively split on the most informative attribute (by Gini / info-gain) until leaves are pure; gives readable if-then rules. Algorithms: ID3, C4.5, CART, CHAID; curb overfitting by pre/post-pruning. Other classifiers: NN, SVM, NB, kNN, GA (Genetic Algorithm), CBR (Case-Based Reasoning). Ensembles combine many models → more robust: bagging (→ Random Forest), boosting, info fusion.

Clustering — unsupervised: group similar records with no target (= segmentation). k-Means: choose k → place k random centres → assign each point to its nearest centre → recompute centres → repeat until stable; pick k by Silhouette / elbow. Also hierarchical (dendrogram), SOM (neural), fuzzy c-means.

Association rules X→Y [S,C] — Support=P(X∩Y) (how often the pair appears); Confidence=P(Y|X) (how often Y follows X); Lift=Conf/P(Y) (>1 = positive association, =1 independent). Apriori finds frequent itemsets ≥ a min-support threshold then builds rules; also Eclat, FP-Growth. Myths/mistakes: DM isn't instant magic — it's a deliberate multistep process; the top mistake is no clear goal / too little time on data prep.

Ch.6 · Predictive Analytics II — Text, Web & Social Media

Key idea · ~85–90% of corporate data is unstructured text; impose structure (term-document matrix) then mine it for topics, sentiment & web/social patterns.

Text analytics = IR (Information Retrieval) + Information Extraction + DM + Web mining (≈ IR + Text Mining). DM vs TM (Text Mining) = structured data (databases) vs unstructured (Word/PDF/web/email); both are semi-automated pattern discovery, but TM must impose structure first. TM applications: information extraction, topic tracking, summarization, categorization, clustering, concept linking, question answering; spam filtering.

Text-mining terms — corpus (document collection); term (word) → concept (related terms); tokenizing (split into words); stop words (remove the/is/and); stemming (crude chop: running→run) vs lemmatization (dictionary form: better→good); n-grams (multi-word terms); case-folding; morphology. TDM (Term-Document Matrix) = terms × documents; cell weighting = binary / term frequency / TF-IDF; the matrix is huge & sparse → shrink by removing rare words, domain-expert filtering, or SVD/LSI.

NLP (Natural Language Processing) — AI + computational linguistics to understand language, going beyond bag-of-words to meaning. Pipeline: tokenize → POS (Part-Of-Speech) tag → parse (syntax) → NER (Named-Entity Recognition) → word-sense disambiguation → coreference. Tasks: machine translation, speech recognition, summarization, question answering, sentiment. Challenges — ambiguity at every level: lexical/word-sense, syntactic (parse), semantic, pragmatic/context; plus imperfect input & segmentation. Resources: WordNet, SentiWordNet.

Text mining — 3 steps: 1 Establish corpus (collect relevant docs; digitize/OCR; clean & normalize) → 2 Create TDM (impose structure; weight cells counts/binary/TF-IDF = Term Frequency–Inverse Document Frequency; cut dimensionality by removing stop/rare words + SVD = Singular Value Decomposition ≈ PCA) → 3 Extract knowledge (classify / cluster / associate / topic / trend).

Topic modeling — unsupervised theme discovery: word-frequency clustering, LSI/LSA (Latent Semantic Indexing/Analysis = TF-IDF + SVD), LDA (Latent Dirichlet Allocation — most popular: a document = mix of topics, a topic = word distribution).

Sentiment analysis (opinion mining) — detect opinions/emotions/attitudes in text: "what do people feel?". Levels: document · sentence · aspect/feature (entity-level — most useful). Output: polarity (positive / negative / neutral) + optional strength & emotion (joy/anger/fear…).

SA — 4 steps: (1) S–O detection (objective fact vs subjective opinion) → (2) N–P polarity (negative/positive) → (3) target/aspect identification (what is the opinion about?) → (4) aggregation (word→sentence→document). Approaches: lexicon-based (sentiment dictionaries, e.g. SentiWordNet — sum polarity scores) vs machine-learning (supervised NB/SVM/logistic; deep learning RNN/LSTM/BERT) vs hybrid. Challenges: negation ("not good"), sarcasm/irony, comparatives, mixed sentiment, domain dependence. Apps: VOC / VOM / VOE (Voice of Customer / Market / Employee), brand monitoring, financial-market & political sentiment, product reviews, government intelligence.

Eg · "screen gorgeous but battery terrible" → screen = Positive, battery = Negative (two aspects, opposite polarity).

DSS & BI foundations

↔Ch.1R1 · A Taxonomy of Decision Support Systems (Alter 1977)

  • Background: in the early 1970s "DSS" was a buzzword pinned on wildly different systems; Alter studied real organisational systems to bring order and a shared vocabulary.
  • Argument: DSS is not homogeneous — systems differ widely in what they do and how directly they shape the decision.
  • Problem: no common language to compare, design or communicate DSS — each was ad-hoc.
  • Methodology: (1) collect 56 real DSS across organisations → (2) describe what each does → (3) rank by degree of action implication (output merely informs vs directly determines the decision) → (4) arrange on a data-oriented ↔ model-oriented spectrum → (5) name 7 types.
  • 7 types: data-oriented — File Drawer Systems (direct data access), Data Analysis Systems (ad-hoc manipulation), Analysis Information Systems (decision DB + small models); model-oriented — Accounting Models (definitional calc, budgets), Representational Models (simulate/estimate, approximate reality), Optimization Models (best solution under constraints), Suggestion Models (recommend a specific action).
  • Compare: data-oriented used hands-on by decision-makers; model-oriented often need staff intermediaries. Accounting models trusted (definitional); representational doubted (approximations).
  • Lessons learned: implementation challenge depends on type • user's role shifts across the taxonomy • model credibility varies • a taxonomy is a design/communication tool • approaches can be combined.
  • One-liner: Alter's 7-type taxonomy (by action implication, data→model) gives a language to design, classify & adopt DSS.

↔Ch.1/4R2 · BI & Its Pivotal Role in Organizational Performance (Moussas 2024)

  • Excluded from Q2 (your presentation reading) — but the strongest backing for Q1.

↔Ch.1/5R9 · Integrated DSS for DM Tools in Economic Intelligence (Salah 2025)

  • Background: economic decision-makers (esp. SMEs) work in volatile environments and need fast, accurate intelligence; the authors build & test a DSS on socio-economic data.
  • Argument: economic intelligence needs fast, accurate DM tools; an integrated DSS combines several.
  • Problem: complex socio-economic data; existing systems process it inaccurately/inefficiently.
  • Methodology: (1) build a socio-economic intelligence database → (2) mine relationships with association rules → (3) add forecasting → (4) train & compare Naïve Bayes, Logistic Regression, incremental C4.5 decision tree → (5) use incremental learning to adapt as data changes.

↔Ch.3R4 · Visual Analytic for Traffic Impact Assessment (Chan 2024)

  • Background: organisations hoard data but BI value stays fragmented across separate warehousing and mining efforts; the review unifies them.
  • Argument: DW + DM together strengthen BI — centralize data, enable historical analysis, discover patterns, predict.
  • Problem: fragmented view of architecture vs techniques.
  • Methodology: literature review — map the DW side and the DM side (below), then join into one BI pipeline (preprocess → train/evaluate → deploy to reports/dashboards/scorecards).
  • DW role: central repository — sources, ETL, storage, metadata, marts, security, governance, query/analysis.
  • DM role: supervised, unsupervised, semi-supervised, text mining, predictive modelling, pattern recognition, visualization.
  • Results / apps: segmentation, market-basket, fraud, predictive analytics, behaviour analysis, healthcare diagnosis, financial risk, operational optimization. Challenges: data quality, scalability, privacy, skilled staff, cost, governance, compliance.
  • Lessons learned: DW = trusted-BI foundation • DM = predictive/actionable insight • ETL + quality first • value rises via dashboards • privacy/governance/compliance major.
  • One-liner: DW provides trusted integrated data; DM extracts predictive patterns — together enabling BI & advantage.

↔Ch.5R6 · Using Data Mining to Predict Student Performance (Zahruddin 2023)

  • Background: at Malaysia's UPNM (national defence university), staff wanted to spot at-risk students early using data already collected at enrolment.
  • Argument: DM predicts student performance from demographic + academic data → find key factors, improve support.
  • Problem/Aim: which demographic factors influence performance, and which algorithm predicts best?
  • Data: gender, age, race, marital status, student status, residential area, geographic location, study status, CGPA.
  • Methodology: KDD — (1) data selection → (2) preparation → (3) cleaning → (4) incorporate prior knowledge → (5) apply models (Neural Network, Decision Tree, k-NN in WEKA) → (6) interpret; compare by RMSE.
  • Results: most influential = gender, age, student status; RMSE — NN 0.0006 (best), k-NN 0.0622, Decision Tree 5.1604 (worst).
  • Compare: NN = best accuracy, harder to explain; k-NN = 2nd, similarity grouping; DT = interpretable but least accurate. (paper wrongly calls k-NN "unsupervised" — it's supervised)
  • Lessons learned: educational DM → early prediction • demographic data reveals patterns • NN accurate but opaque • DT interpretable not always accurate • analytics → better support strategies.
  • One-liner: KDD classification on demographic data predicts performance; Neural Networks most accurate (RMSE 0.0006) for early at-risk support.

↔Ch.5R7 · Customer Segmentation using K-Means & DBSCAN (William 2025)

  • Background: an online fashion store ("IXYZ") had piles of transactions but no customer segments to target marketing.
  • Argument: segmentation → fashion e-commerce understands buying behaviour for targeted marketing; K-Means beat DBSCAN.
  • Problem: turn raw transactions into meaningful, marketable customer groups.
  • Methodology: CRISP-DM — (1) collect 10,845×49 transactions → (2) clean + build RFM features → (3) StandardScaler + PCA → (4) run K-Means (sweep k) & DBSCAN → (5) evaluate Silhouette ↑ / Davies-Bouldin ↓ → (6) profile segments → (7) deploy (FastAPI + React/TypeScript).
  • Results: K-Means best at k=7 (Silhouette 0.549, DBI 0.593); DBSCAN worse (0.2906 / 0.92); deployed FastAPI + React/TypeScript.
  • Compare: K-Means suits centroid groups; DBSCAN good for outliers, not best here; PCA + scaling improve quality.
  • Lessons learned: segmentation reveals buying patterns • K-Means for centroid groups • DBSCAN for outliers • PCA + scaling improve clustering • dashboards → marketing decisions.
  • One-liner: CRISP-DM clustering (K-Means k=7 > DBSCAN) yields 7 actionable fashion segments, deployed as a dashboard.

↔Ch.5R8 · Patients at Higher CV Risk for Prescribed Stimulants (Yan 2025)

  • Background: adult ADHD stimulant prescriptions are rising; clinicians worry about heart risk but prior evidence is mixed, so the authors mine health records to find who is actually at risk.
  • Argument: predictive analytics + DM find which patients face higher cardiovascular (CV) risk on stimulants — risk is patient-specific.
  • Problem: which patient subgroups face higher CV risk on stimulants?
  • Data: EHR (Electronic Health Records, TriNetX; ADHD adults 2010–20); cohort 219,965 (102,138 stimulant users), 4 age groups.
  • Methodology: (1) build cohort from EHR → (2) split by age group → (3) train ML to predict 1-year CV risk → (4) take the highest-25% added-risk patients → (5) run Association Rule Mining (ARM) to find common comorbidity combinations.
  • Results: AUROC 0.77–0.84; risk categories = existing CV risk factors, prior CV events, substance-use disorders, psychological disorders → comorbidity watch-list.
  • Lessons learned: models reveal patient-specific (not avg) risk • EHR → safer medication decisions • ARM finds risky combos • risk → practical watch-list • DM strengthens healthcare DSS.
  • One-liner: ML + ARM on 219,965 EHRs (AUROC 0.77–0.84) builds a comorbidity watch-list for safer stimulant prescribing.

Text / web / social (unstructured)

↔Ch.6R10 · Text-Based Content Analysis on Social Media (Topic Modelling) (Buana 2024)

  • Background: Instagram marketing is image-heavy, but the useful text sits inside the images — invisible to normal analytics that only read likes/captions.
  • Argument: social media analytics (SMA) supports digital marketing by discovering topics from text inside Instagram posts.
  • Problem: marketers need topic ideas, but tools analyse metadata (likes/comments) and ignore image text.
  • Methodology: (1) collect posts (content text, captions, likes, comments, upload time, category) → (2) OCR (Optical Character Recognition) extracts text from images → (3) clean text → (4) LDA finds dominant topics → (5) wrap as an SMA tool → (6) test with expert users.
  • Results: expert-tested — 93% could find topic references, 85% usable despite difficulty (UX needs work).
  • Lessons learned: SMA → content ideation • OCR makes image text readable • LDA finds hidden themes • metadata alone insufficient • UX matters for adoption.
  • One-liner: OCR + LDA turn Instagram image text into topic insights (93% usable) for content/marketing planning.

↔Ch.6R11 · Social Intelligence & Web Mining for Renewable-Energy Insights (Mwafaq 2025)

  • Background: renewable-energy policy depends on public opinion, which lives scattered across social media, forums & news; the authors mine it at scale.
  • Argument: social intelligence + web mining analyse online opinion on renewable energy → policy-level insight.
  • Problem: adoption depends on opinion, awareness, trust & regional readiness — not just technology.
  • Methodology: (1) crawl 1.5M → 900k social/forum/news items over 12 months → (2) clean → (3) RoBERTa transformer sentiment + LDA themes → (4) multivariate/ridge regression of sentiment → adoption.
  • Results: 58% positive (EU/N.America lead); top themes = economic benefit + technology advances; predicts regional adoption (Europe ≈15.5%).
  • Lessons learned: web mining scales opinion analysis • transformer sentiment (RoBERTa) > lexicon • topic modelling surfaces themes • sentiment can predict adoption • regional differences matter for policy.
  • One-liner: RoBERTa sentiment + web mining + regression on ~900k posts → 58% positive opinion, predicting regional renewable adoption.

↔Ch.6R12 · Sentiment Analysis & CEEMDAN for Consumer-Group Preferences (Wang 2026)

  • Background: hotels get thousands of online reviews but can't easily turn emotional, ambiguous, time-varying text into stable consumer preferences.
  • Argument: reviews hide consumer preferences (unstructured, emotional, time-dependent); combine NLP + decision modelling to learn them.
  • Problem: hotel reviews are scattered, implicit, ambiguous & change over time.
  • Methodology: (1) collect Ctrip reviews 14,133 → 10,505 after cleaning → (2) BERT-Base-Chinese fine-grained (aspect) sentiment (F1 0.83) → (3) BERTopic extracts 20 topics → (4) CEEMDAN separates short-term noise from long-term trend → (5) UTA preference model derives criteria weights, thresholds & value systems per consumer group.
  • Results: interpretable group preferences; identifies each group's priority factors; beats LSTM (0.761) & RF (0.568); manager decision support.
  • Lessons learned: fine-grained sentiment > pos/neg • preferences change over time • CEEMDAN separates trend from noise • NLP + decision modelling → interpretability • review mining → service/marketing/resource decisions.
  • One-liner: BERT + CEEMDAN + UTA mine hotel reviews into stable, interpretable consumer-group preferences for evidence-based management.

Formulas, Metrics & Quick-Pick

Classification — Accuracy=(TP+TN)/all; Precision=TP/(TP+FP); Recall/TPR=TP/(TP+FN); TNR=TN/(TN+FP); F1=2·P·R/(P+R) (best for imbalance); AUC 0.5=random, 1=perfect.

Regression — y=β₀+β₁x; fit=OLS; quality=R², p-value, RMSE/MAD/MSE. Logistic 1/(1+e^−(β₀+β₁x)) → P(class). Tree split = Gini / info-gain(entropy); ensembles = bagging(→RF) / boosting.

Association — Support=P(X∩Y); Confidence=P(Y|X); Lift=Conf/P(Y) (>1 positive, =1 independent, <1 negative). Clustering — Silhouette ↑ better, Davies-Bouldin ↓ better.

Text — TF-IDF=TF×log(N/dfₜ) (high = frequent here, rare overall); reduce TDM via SVD. DPMO=defects/(units×opportunities)×10⁶; Six Sigma=3.4 DPMO.

Which technique? — categorical target+labels → classification (tree/NN/SVM/NB); numeric target → regression; no target, find groups → clustering (k-means); items bought together → association (Apriori); unstructured text → text mining/LDA/sentiment; "what happened?" → descriptive/viz; "best action?" → prescriptive/optimization.
